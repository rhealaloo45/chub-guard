# System Overview: chub Deprecation Guard

This document explains the internal architecture, design decisions, and execution flow of the `chub_guard.py` CLI tool.

## Core Objective
The goal of the tool is to bridge the gap between static analysis and rapidly evolving third-party AI SDKs. Standard tools like `ruff` or `flake8` only understand Python standard library deprecations. The `chub Deprecation Guard` introduces a mechanism to dynamically inject AI SDK deprecation rules into the linting pipeline using live documentation.

## Architecture Flow

The tool executes via a 6-step pipeline during the `pre-commit` stage:

### 1. File Collection & AST Discovery
When triggered by `pre-commit`, the script receives a list of staged `.py` files. It uses Python's built-in `ast` (Abstract Syntax Tree) module to safely parse the source code without executing it. 
It walks the AST to collect all `import` and `from ... import` statements to build a mapping of which files depend on which third-party modules.

### 2. Registry Resolution
The script cross-references the discovered modules against `.chub-docs/registry.json`. 
This registry maps Python package names (e.g., `langchain`, `openai`, `mistralai`) to their official `chub` documentation IDs (e.g., `langchain/package`, `openai/package`, `mistralai/package`).

### 3. Dynamic Documentation Fetching & Caching
For any matched SDKs, the tool attempts to fetch the latest Markdown documentation via the `chub` CLI:
- **Cache Layer:** It first checks `.chub-docs/`. If the doc was downloaded within the last 24 hours, it skips the network request.
- **CLI Fetch:** If the cache is cold, it runs `chub get <doc_id> --lang python --output <path>`.
- **GitHub Raw Fallback:** If the `chub` CLI is unavailable or fails (a common issue with absolute paths on Windows), it gracefully falls back to fetching the raw `DOC.md` directly from the `andrewyng/context-hub` GitHub repository via `urllib`.

### 4. Hybrid Deprecation Analysis
This is the core innovation of the tool. It uses a two-pronged approach to find deprecations:

**A. Targeted AST Synthesis (The Failsafe)**
Critical ecosystem-breaking changes are caught explicitly via AST traversal. This ensures zero false positives for the most common errors generated by AI coding assistants.
*Examples caught here:* `openai.ChatCompletion.create()`, `import google.generativeai`, `Anthropic().completions.create()`.

**B. Dynamic Markdown Parsing (The Future-Proof Layer)**
The tool dynamically reads the downloaded `chub` markdown file. It uses Regular Expressions to scan for code snippets wrapped in backticks (e.g., \`MistralClient\`) that appear on lines containing negative keywords like `"incorrect"`, `"deprecated"`, `"legacy"`, or `"do not use"`.
It then scans the staged Python files for these dynamically extracted patterns. If it finds a match, it flags it. This allows the tool to catch new deprecations (like `HuggingFaceHub` or `SequentialChain`) the moment the `chub` maintainers update the documentation, without requiring an update to the Python script.

### 5. Standard Linting Integration
The tool executes `ruff check --select UP --output-format json` to catch standard Python pyupgrade (`UP`) violations natively. 
It merges its dynamically synthesized AI SDK violations (labeled as `UP035`) into the `ruff` JSON output.

### 6. Hint Extraction & Rich UI Rendering
If violations are found, the tool needs to provide a solution. It parses the cached `chub` markdown document, searching for the `"## Usage"` or `"## Core Usage"` headers. It extracts the first valid Python code block it finds beneath those headers.
Finally, it uses the `rich` library to render a beautiful terminal UI, displaying the file name, the deprecated code, the error message, and the extracted "Current API" usage hint to guide the developer.

## Edge Cases Handled

1. **Duplicate Blocking:** The synthesis engine checks if a violation has already been logged for a specific line/module to prevent spamming the user.
2. **False Positive Suppression:** Before flagging a synthesized violation, the tool checks the raw source line for a `# noqa: UP...` comment. If present, the violation is silently dropped.
3. **Graceful Degradation:** If `chub` is completely unreachable and the GitHub fallback fails, the tool emits a yellow warning but continues to run standard `ruff` linting. It will not block a commit simply because the internet is down.
4. **Syntax Errors:** If a staged Python file has a fatal syntax error, the AST parser emits a warning and skips the file rather than crashing the entire hook.
